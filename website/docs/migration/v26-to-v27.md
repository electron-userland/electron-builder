# Migrating from v26 to v27

v27 migrates the entire electron-builder package ecosystem to native ES modules (ESM) and raises the minimum Node.js version to **22.12.0**.

**Most projects need only a Node.js version bump.** The `build()` API, all configuration options, and all exported types are unchanged. CJS `require()` continues to work without any code changes on supported Node.js versions.

---

## Breaking changes at a glance

| Change | Action required |
|--------|----------------|
| Node.js >=22.12.0 required | Update your runtime and CI node version |
| All packages are native ESM | None ŌĆö `require()` still works on Node >=22.12 |
| `electronCompile` config option removed | Remove from build config; migrate off `electron-compile` |
| `electron-forge-maker-*` are now ESM | None ŌĆö same API, same export shape |
| Toolset env-var overrides removed | Replace `APPIMAGE_TOOLS_PATH`, `ELECTRON_BUILDER_NSIS_DIR`, etc. with `toolsets.X: { url, checksum }` |
| Toolset default versions bumped | Optionally pin to `"0.0.0"` to restore legacy bundle if needed |
| `framework`, `nodeVersion`, `launchUiVersion` removed | Remove from build config (ProtonFramework/LibUiFramework support is gone) |
| ProtonFramework and LibUiFramework removed | Migrate to an Electron-based build setup |
| `snap` key deprecated | Rename to `snapcraft` and add an explicit `base` |

---

## 1. Update Node.js to >=22.12.0

v27 requires **Node.js 22.12.0 or later**. This is the version where Node's [`require(esm)`](https://nodejs.org/en/blog/release/v22.12.0) support was stabilized (no flags needed), which allows both CJS and ESM consumers to use these packages without any code changes.

Update your local environment:
```bash
# nvm
nvm install 22 && nvm use 22

# fnm
fnm install 22 && fnm use 22
```

Update CI (GitHub Actions):
```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '22'
```

Update your `package.json` engines field if you declare one:
```json
{
  "engines": {
    "node": ">=22.12.0"
  }
}
```

---

## 2. ESM output ŌĆö what changes for your project

electron-builder packages now ship as native ES modules. On **Node >=22.12.0** both styles continue to work:

**CJS `require()` ŌĆö still works**
```js
const { build, Platform } = require("electron-builder")
```

**ESM `import` ŌĆö now the preferred style**
```js
import { build, Platform } from "electron-builder"
```

**TypeScript**
```ts
import { build, Configuration, Platform } from "electron-builder"

const config: Configuration = {
  appId: "com.example.myapp",
  mac: { target: "dmg" },
}

await build({ targets: Platform.MAC.createTarget(), config })
```

If your project uses `"type": "module"` or ESM imports, no changes are needed. If it uses CJS on Node >=22.12, `require()` continues to work as before.

---

## 3. Remove `electronCompile` usage

The `electronCompile` configuration option has been **removed**. `electron-compile` is an unmaintained library with no releases since 2019.

**Remove from your configuration:**
```json5
{
  "build": {
    "electronCompile": true  // ŌåÉ delete this line
  }
}
```

If your project relies on `electron-compile` for source compilation, migrate to a modern build tool before upgrading to v27:

- **[electron-vite](https://electron-vite.org/)** ŌĆö recommended; fast, first-class ESM and Electron integration
- **[esbuild](https://esbuild.github.io/)** ŌĆö extremely fast, minimal configuration
- **[webpack](https://webpack.electron.build/)** ŌĆö mature, widely used, `electron-webpack` for Electron-specific setup

Quick start with electron-vite:
```bash
npm create electron-vite@latest my-app
```

---

## 4. `electron-forge-maker-*` plugins

The four Forge maker plugins (`electron-forge-maker-appimage`, `electron-forge-maker-nsis`, `electron-forge-maker-nsis-web`, `electron-forge-maker-snap`) are now native ES modules. The public API is identical ŌĆö Electron Forge loads them via dynamic `import()` internally, so no config changes are required.

---

## 5. Programmatic API ŌĆö no changes

The `build()` function signature, all configuration options, and all exported types are unchanged:

```js
// v26 CJS
const { build } = require("electron-builder")

// v27 CJS ŌĆö still works on Node >=22.12
const { build } = require("electron-builder")

// v27 ESM ŌĆö preferred
import { build } from "electron-builder"
```


## 8. Toolset defaults updated

All built-in binary toolsets have been upgraded to modern default versions in v27. The new defaults include:

| Toolset | v26 default | v27 default | Key improvement |
|---------|-------------|-------------|-----------------|
| `winCodeSign` | `0.0.0` (winCodeSign 2.6.0) | `"1.1.0"` | Windows Kits 10.0.26100.0; modern `signtool.exe` and `osslsigncode` |
| `nsis` | `0.0.0` (NSIS 3.0.4.1, split bundle) | `"1.2.1"` | NSIS 3.12; unified single-archive bundle |
| `appimage` | `0.0.0` (FUSE2 runtime) | `"1.0.3"` | FUSE3-compatible static runtime (runs without host FUSE install) |
| `wine` | `0.0.0` (Wine 4.0.1, macOS only) | `"1.0.1"` | Wine 11.0; macOS + Linux; ia32 via WoW64 |

**No action required** for most projects ŌĆö the new bundles are drop-in replacements and produce identical output.

**If you hit a regression** introduced by a newer bundle, you can pin back to the legacy bundle by setting the toolset version to `"0.0.0"` in your build config:

```json5
{
  "build": {
    "toolsets": {
      "winCodeSign": "0.0.0",   // restore winCodeSign 2.6.0
      "nsis": "0.0.0",          // restore NSIS 3.0.4.1
      "appimage": "0.0.0",      // restore FUSE2 AppImage runtime
      "wine": "0.0.0"           // restore Wine 4.0.1 (macOS only)
    }
  }
}
```

## 9. Toolset env-var overrides replaced by `ToolsetCustom`

**This is a breaking change if you used env-var toolset overrides.**

The following environment variables that previously let you point electron-builder at a local binary bundle have been **removed**:

| Removed env var | Toolset it controlled |
|---|---|
| `APPIMAGE_TOOLS_PATH` | AppImage build tools (`mksquashfs`, runtime) |
| `LINUX_TOOLS_MAC_PATH` | Linux-tools-mac bundle (`ar`, `lzip`, `gtar`) |
| `CUSTOM_FPM_PATH` | FPM executable |
| `ELECTRON_BUILDER_NSIS_DIR` | NSIS compiler bundle directory |
| `ELECTRON_BUILDER_NSIS_RESOURCES_DIR` | NSIS resources/plugins directory |
| `CUSTOM_NSIS_RESOURCES` | Alternate NSIS resources bundle |
| `ELECTRON_BUILDER_WINE_TOOLSET_DIR` | Wine bundle directory |

### Migration: use `ToolsetCustom`

Replace env-var overrides with the new `ToolsetCustom` config object on the relevant `toolsets` key. The `url` accepts an `https://` URL (downloaded and cached automatically) or a `file://` path (used as-is).

**Remote bundle (URL)**
```json5
{
  "build": {
    "toolsets": {
      "nsis": {
        "url": "https://example.com/my-nsis-bundle-1.0.tar.gz",
        "checksum": "sha256:abc123...",
        "version": "my-custom-1.0"
      }
    }
  }
}
```

**Local archive**
```json5
{
  "build": {
    "toolsets": {
      "wine": {
        "url": "file:///path/to/my-wine-bundle.tar.gz",
        "checksum": "sha256:def456..."
      }
    }
  }
}
```

**Local directory (no checksum required)**
```json5
{
  "build": {
    "toolsets": {
      "appimage": {
        "url": "file:///path/to/my-appimage-tools-dir"
      }
    }
  }
}
```

The bundle must mirror the directory layout of the corresponding built-in bundle. See [electron-builder-binaries/packages](https://github.com/electron-userland/electron-builder-binaries/tree/master/packages) for the expected structure.

Supported archive formats: `.zip`, `.7z`, `.tar.gz`, `.tar.xz`. **Exception for `sevenZip`**: because 7-Zip is used to extract `.7z` and `.tar.xz` archives, a custom `sevenZip` bundle can only be supplied as a `.tar.gz`, `.zip`, or bare `file://` directory.

## 10. `framework`, `nodeVersion`, `launchUiVersion` config options removed

The `framework`, `nodeVersion`, and `launchUiVersion` build configuration options have been **removed**. These were only used by the ProtonFramework and LibUiFramework integrations, which are no longer supported.

**Remove from your configuration if present:**
```json5
{
  "build": {
    "framework": "proton",       // ŌåÉ delete
    "nodeVersion": "current",    // ŌåÉ delete
    "launchUiVersion": "0.1.1"  // ŌåÉ delete
  }
}
```

If your project targets proton-native or libui, you will need to set up your own build pipeline ŌĆö electron-builder now exclusively supports Electron-based applications.

---

## 11. `snap` key deprecated ŌĆö migrate to `snapcraft`

The top-level `snap` configuration key is deprecated in v27. Use `snapcraft` instead, which is identical in structure but requires an explicit `base` and supports the newer `core24` base.

**Before:**
```json5
{
  "build": {
    "snap": {
      "summary": "My app"
    }
  }
}
```

**After:**
```json5
{
  "build": {
    "snapcraft": {
      "base": "core24",
      "core24": {
        "summary": "My app"
      }
    }
  }
}
```

When both `snapcraft` and `snap` are present, `snapcraft` takes precedence. Remove the `snap` key to silence the deprecation warning.

## 12. Plugin and custom-target authors ŌĆö `PlatformPackager` API changes

> **This section only affects you if you import from `app-builder-lib` and access the `packager` or `platformPackager` objects directly.** Standard project configurations are unaffected.

### `info: Packager` is now `@deprecated`

`PlatformPackager` has always exposed a public `info: Packager` field that gave access to the whole-build orchestrator. In v27 this field is **deprecated** and will become `protected` in a future major release.

TypeScript will emit a `TS6385` hint anywhere your code chains through `.info.`:

```ts
// deprecated ŌĆö produces a TS6385 hint
const tmpDir = packager.info.tempDirManager
const meta = packager.info.metadata
await packager.info.emitArtifactCreated(event)
```

### New pass-through getters and methods

All commonly-needed properties are now directly accessible on `PlatformPackager` itself. Migrate by dropping the `.info.` chain:

| Before (deprecated) | After |
|---|---|
| `packager.info.tempDirManager` | `packager.tempDirManager` |
| `packager.info.metadata` | `packager.metadata` |
| `packager.info.framework` | `packager.framework` |
| `packager.info.cancellationToken` | `packager.cancellationToken` |
| `packager.info.repositoryInfo` | `packager.repositoryInfo` |
| `packager.info.relativeBuildResourcesDirname` | `packager.relativeBuildResourcesDirname` |
| `packager.info.stageDirPathCustomizer` | `packager.stageDirPathCustomizer` |
| `packager.info.areNodeModulesHandledExternally` | `packager.areNodeModulesHandledExternally` |
| `packager.info.isPrepackedAppAsar` | `packager.isPrepackedAppAsar` |
| `packager.info.appDir` | `packager.appDir` |
| `packager.info.getWorkspaceRoot()` | `packager.getWorkspaceRoot()` |
| `packager.info.emitArtifactBuildStarted(e)` | `packager.emitArtifactBuildStarted(e)` |
| `packager.info.emitArtifactBuildCompleted(e)` | `packager.emitArtifactBuildCompleted(e)` |
| `packager.info.emitArtifactCreated(e)` | `packager.emitArtifactCreated(e)` |
| `packager.info.emitMsiProjectCreated(p)` | `packager.emitMsiProjectCreated(p)` |
| `packager.info.emitAppxManifestCreated(p)` | `packager.emitAppxManifestCreated(p)` |

The getters already on `PlatformPackager` in v26 are unchanged (`config`, `projectDir`, `buildResourcesDir`, `packagerOptions`, `appInfo`, `debugLogger`).
