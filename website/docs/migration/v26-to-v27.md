# Migrating from v26 to v27

v27 migrates the entire electron-builder package ecosystem to native ES modules (ESM), raises the minimum Node.js version to **22.12.0**, and removes a large set of long-deprecated configuration options and behaviors.

**Most projects need only a Node.js version bump.** The `build()` API and all exported types are unchanged unless you were using one of the deprecated options listed below.

---

## Breaking changes at a glance

| Change | Action required |
|--------|----------------|
| Node.js >=22.12.0 required | Update your runtime and CI node version |
| All packages are native ESM | None — `require()` still works on Node >=22.12 |
| `electronCompile` config option removed | Remove from build config; migrate off `electron-compile` |
| `electron-forge-maker-*` are now ESM | None — same API, same export shape |
| Implicit `--publish` removed | Pass `--publish` explicitly |
| `snap` config key removed | Use `snapcraft` with an explicit `base` field |
| `GithubOptions.vPrefixedTagName` removed | Use `tagNamePrefix` instead |
| `GitlabOptions.vPrefixedTagName` removed | Tags now always use the `v` prefix |
| `framework`, `nodeVersion`, `launchUiVersion` removed | Electron is the only supported framework |
| `devMetadata`, `extraMetadata` in `PackagerOptions` removed | Use `config` / `config.extraMetadata` instead |
| `--em.build`, `--em.directories` CLI flags removed | Use `-c` / `-c.directories` instead |
| `npmSkipBuildFromSource` removed | Use `buildDependenciesFromSource` |
| `appImage.systemIntegration` removed | Handled automatically via AppImageLauncher |
| `asar-unpack`, `asar-unpack-dir`, `asar.unpackDir`, `asar.unpack` removed | Use `asarUnpack` |
| `directories` at package.json root removed | Specify under `build.directories` |

---

## 1. Update Node.js to >=22.12.0

v27 requires **Node.js 22.12.0 or later**. This is the version where Node's [`require(esm)`](https://nodejs.org/en/blog/release/v22.12.0) support was stabilized (no flags needed), which allows both CJS and ESM consumers to use these packages without any code changes.

Update your local environment:
```bash
# nvm
nvm install 22 && nvm use 22

# fnm
fnm install 22 && fnm use 22
```

Update CI (GitHub Actions):
```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '22'
```

Update your `package.json` engines field if you declare one:
```json
{
  "engines": {
    "node": ">=22.12.0"
  }
}
```

---

## 2. ESM output — what changes for your project

electron-builder packages now ship as native ES modules. On **Node >=22.12.0** both styles continue to work:

**CJS `require()` — still works**
```js
const { build, Platform } = require("electron-builder")
```

**ESM `import` — now the preferred style**
```js
import { build, Platform } from "electron-builder"
```

**TypeScript**
```ts
import { build, Configuration, Platform } from "electron-builder"

const config: Configuration = {
  appId: "com.example.myapp",
  mac: { target: "dmg" },
}

await build({ targets: Platform.MAC.createTarget(), config })
```

If your project uses `"type": "module"` or ESM imports, no changes are needed. If it uses CJS on Node >=22.12, `require()` continues to work as before.

---

## 3. Remove `electronCompile` usage

The `electronCompile` configuration option has been **removed**. `electron-compile` is an unmaintained library with no releases since 2019.

**Remove from your configuration:**
```json5
{
  "build": {
    "electronCompile": true  // ← delete this line
  }
}
```

If your project relies on `electron-compile` for source compilation, migrate to a modern build tool before upgrading to v27:

- **[electron-vite](https://electron-vite.org/)** — recommended; fast, first-class ESM and Electron integration
- **[esbuild](https://esbuild.github.io/)** — extremely fast, minimal configuration
- **[webpack](https://webpack.electron.build/)** — mature, widely used, `electron-webpack` for Electron-specific setup

Quick start with electron-vite:
```bash
npm create electron-vite@latest my-app
```

---

## 4. `electron-forge-maker-*` plugins

The four Forge maker plugins (`electron-forge-maker-appimage`, `electron-forge-maker-nsis`, `electron-forge-maker-nsis-web`, `electron-forge-maker-snap`) are now native ES modules. The public API is identical — Electron Forge loads them via dynamic `import()` internally, so no config changes are required.

---

## 5. Implicit `--publish` removed

electron-builder previously auto-published when certain conditions were detected at build time:

- npm lifecycle event was `"release"` → published with `"always"`
- A git tag was present in CI → published with `"onTag"`
- A CI environment was detected → published with `"onTagOrDraft"`

**This behavior is removed in v27.** Builds never auto-publish. You must pass `--publish` explicitly:

```bash
# publish on every build
electron-builder --publish always

# publish only when a git tag is present
electron-builder --publish onTag

# publish when a git tag is present or a draft release exists
electron-builder --publish onTagOrDraft

# never publish (explicit opt-out, same as before)
electron-builder --publish never
```

Or in your `package.json` scripts:
```json
{
  "scripts": {
    "release": "electron-builder --publish always"
  }
}
```

---

## 6. `snap` config key removed — use `snapcraft`

The top-level `snap` configuration key is removed. Use `snapcraft` with an explicit `base` field instead.

**Before:**
```json5
{
  "build": {
    "snap": {
      "summary": "My app",
      "base": "core22"
    }
  }
}
```

**After:**
```json5
{
  "build": {
    "snapcraft": {
      "base": "core22",
      "core22": {
        "summary": "My app"
      }
    }
  }
}
```

Supported `base` values: `"core18"`, `"core20"`, `"core22"`, `"core24"`. For new projects, `"core24"` is recommended (requires Electron 25+; Electron 28+ highly recommended).

---

## 7. `GithubOptions.vPrefixedTagName` removed — use `tagNamePrefix`

The `vPrefixedTagName` boolean on `GithubOptions` is removed. Use `tagNamePrefix` to control the tag prefix.

**Before:**
```json5
{
  "build": {
    "publish": {
      "provider": "github",
      "vPrefixedTagName": false
    }
  }
}
```

**After:**
```json5
{
  "build": {
    "publish": {
      "provider": "github",
      "tagNamePrefix": ""      // empty string = no prefix
    }
  }
}
```

To keep the default `v` prefix, simply remove `vPrefixedTagName` — `tagNamePrefix` defaults to `"v"`.

### GitLab tag prefix

`vPrefixedTagName` on `GitlabOptions` is also removed. GitLab releases now always use the `v`-prefixed tag (e.g. `v1.2.3`). There is no replacement option; update your release pipeline if your GitLab tags do not follow this convention.

---

## 8. `framework`, `nodeVersion`, `launchUiVersion` removed

The `framework`, `nodeVersion`, and `launchUiVersion` configuration fields are removed. Only Electron is supported as a target framework — `proton`/`proton-native` and `libui` support has been removed.

**Remove from your configuration:**
```json5
{
  "build": {
    "framework": "electron",       // ← delete; "electron" was and remains the default
    "nodeVersion": "current",      // ← delete; no effect
    "launchUiVersion": "0.1.0"    // ← delete; no effect
  }
}
```

---

## 9. Removed `PackagerOptions` fields: `devMetadata`, `extraMetadata`

The `devMetadata` and `extraMetadata` fields in the programmatic `PackagerOptions` API are removed (they have thrown `InvalidConfigurationError` since v22).

**Before:**
```ts
await build({
  targets: Platform.MAC.createTarget(),
  devMetadata: { ... },    // ← throws in v22–v26, removed in v27
  extraMetadata: { ... },  // ← throws in v22–v26, removed in v27
})
```

**After:**
```ts
await build({
  targets: Platform.MAC.createTarget(),
  config: {
    extraMetadata: { ... },
  },
})
```

---

## 10. Removed CLI flags: `--em.build`, `--em.directories`

The `--em.build` and `--em.directories` CLI flags are removed (they have thrown since v22).

| Removed flag | Replacement |
|---|---|
| `--em.build` | `-c` (pass build config inline) |
| `--em.directories` | `-c.directories` |

---

## 11. Removed options: `npmSkipBuildFromSource`, `appImage.systemIntegration`

- **`npmSkipBuildFromSource`** — removed. Use `buildDependenciesFromSource` instead.
- **`appImage.systemIntegration`** — removed. Desktop integration is handled automatically by [AppImageLauncher](https://github.com/TheAssassin/AppImageLauncher).

---

## 12. Removed legacy asar options

The following asar-related config keys are removed (they have thrown since v22). Use `asarUnpack` in all cases.

| Removed key | Replacement |
|---|---|
| `asar-unpack` | `asarUnpack` |
| `asar-unpack-dir` | `asarUnpack` |
| `asar.unpackDir` | `asarUnpack` |
| `asar.unpack` | `asarUnpack` |

**Example:**
```json5
{
  "build": {
    "asarUnpack": ["**/*.node", "resources/**"]
  }
}
```

---

## 13. `directories` must be under `build`, not at the package.json root

Specifying `directories` at the root of `package.json` is removed. Move it under the `build` key.

**Before:**
```json
{
  "name": "my-app",
  "directories": { "output": "dist" }
}
```

**After:**
```json
{
  "name": "my-app",
  "build": {
    "directories": { "output": "dist" }
  }
}
```

---

## 14. Programmatic API — no other changes

The `build()` function signature and all other configuration options and exported types are unchanged.

---

## 15. TypeScript `moduleResolution` settings

electron-builder v27 packages include `exports` maps and full TypeScript declarations. The following `tsconfig.json` settings all work:

| `moduleResolution` | Works |
|---|---|
| `"node"` (legacy) | ✓ |
| `"node16"` / `"nodenext"` | ✓ |
| `"bundler"` | ✓ (recommended) |

## 16. CI and Docker environments

Confirm your CI and Docker images run Node.js 22.12 or later.

**GitHub Actions**
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
```

**Docker**
```dockerfile
FROM node:22-bookworm-slim
```

electron-builder's own Docker images for Linux builds have been updated to Node 22.

## 8. Plugin and custom-target authors — `PlatformPackager` API changes

> **This section only affects you if you import from `app-builder-lib` and access the `packager` or `platformPackager` objects directly.** Standard project configurations are unaffected.

### `info: Packager` is now `@deprecated`

`PlatformPackager` has always exposed a public `info: Packager` field that gave access to the whole-build orchestrator. In v27 this field is **deprecated** and will become `protected` in a future major release.

TypeScript will emit a `TS6385` hint anywhere your code chains through `.info.`:

```ts
// deprecated — produces a TS6385 hint
const tmpDir = packager.info.tempDirManager
const meta = packager.info.metadata
await packager.info.emitArtifactCreated(event)
```

### New pass-through getters and methods

All commonly-needed properties are now directly accessible on `PlatformPackager` itself. Migrate by dropping the `.info.` chain:

| Before (deprecated) | After |
|---|---|
| `packager.info.tempDirManager` | `packager.tempDirManager` |
| `packager.info.metadata` | `packager.metadata` |
| `packager.info.framework` | `packager.framework` |
| `packager.info.cancellationToken` | `packager.cancellationToken` |
| `packager.info.repositoryInfo` | `packager.repositoryInfo` |
| `packager.info.relativeBuildResourcesDirname` | `packager.relativeBuildResourcesDirname` |
| `packager.info.stageDirPathCustomizer` | `packager.stageDirPathCustomizer` |
| `packager.info.areNodeModulesHandledExternally` | `packager.areNodeModulesHandledExternally` |
| `packager.info.isPrepackedAppAsar` | `packager.isPrepackedAppAsar` |
| `packager.info.appDir` | `packager.appDir` |
| `packager.info.getWorkspaceRoot()` | `packager.getWorkspaceRoot()` |
| `packager.info.emitArtifactBuildStarted(e)` | `packager.emitArtifactBuildStarted(e)` |
| `packager.info.emitArtifactBuildCompleted(e)` | `packager.emitArtifactBuildCompleted(e)` |
| `packager.info.emitArtifactCreated(e)` | `packager.emitArtifactCreated(e)` |
| `packager.info.emitMsiProjectCreated(p)` | `packager.emitMsiProjectCreated(p)` |
| `packager.info.emitAppxManifestCreated(p)` | `packager.emitAppxManifestCreated(p)` |

The getters already on `PlatformPackager` in v26 are unchanged (`config`, `projectDir`, `buildResourcesDir`, `packagerOptions`, `appInfo`, `platformOptions`, `debugLogger`).

### `platformSpecificBuildOptions` renamed to `platformOptions`

The internal field `platformSpecificBuildOptions` is now `protected`. External code that accessed it directly must use the public getter instead:

```ts
// before
const opts = packager.platformSpecificBuildOptions

// after
const opts = packager.platformOptions
```

### New `getOptionsForTarget<T>(key)` helper

Targets that previously built their options via manual `deepAssign` calls can now use:

```ts
readonly options = this.packager.getOptionsForTarget<MyTargetOptions>("myTarget")
```

This performs the standard platform-options → target-config merge in one call.
