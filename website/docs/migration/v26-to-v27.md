# Migrating from v26 to v27

v27 migrates the entire electron-builder package ecosystem to native ES modules (ESM) and raises the minimum Node.js version to **22.12.0**.

**Most projects need only a Node.js version bump.** The `build()` API, all configuration options, and all exported types are unchanged. CJS `require()` continues to work without any code changes on supported Node.js versions.

---

## Automated migration tool

Run the built-in command to automatically apply the config-level breaking changes to your project:

```bash
electron-builder migrate-schema
```

This rewrites your `electron-builder.json`, `electron-builder.yml`, `package.json` (build key), or any other static config format in place. Use `--dry-run` (`-n`) to preview changes without writing:

```bash
electron-builder migrate-schema --dry-run
```

Pass `--config <path>` to point at a non-default config file, or `--project-dir <dir>` to specify the project root (default: current directory).

> **Note:** Programmatic configs (`.js`, `.ts`, `.cjs`, `.mjs`) and TOML configs cannot be auto-rewritten. The command prints the required manual steps instead.

## Breaking changes at a glance

| Change | Auto-migrated | Action required |
|--------|:---:|----------------|
| Node.js >=22.12.0 required | — | Update your runtime and CI node version |
| All packages are native ESM | — | None — `require()` still works on Node >=22.12 |
| `electronCompile` config option removed | ✓ | Remove from build config; migrate off `electron-compile` |
| `electron-forge-maker-*` are now ESM | — | None — same API, same export shape |
| Implicit `--publish` removed | — | Pass `--publish` explicitly |
| `snap` config key removed | — | Use `snapcraft` with an explicit `base` field |
| `GithubOptions.vPrefixedTagName` removed | ✓ | Use `tagNamePrefix` instead |
| `GitlabOptions.vPrefixedTagName` removed | ✓ | Removed automatically |
| `framework`, `nodeVersion`, `launchUiVersion` removed | ✓ | Removed automatically |
| ProtonFramework and LibUiFramework removed | — | Migrate to an Electron-based build setup |
| `devMetadata`, `extraMetadata` in `PackagerOptions` removed | — | Use `config` / `config.extraMetadata` instead |
| `--em.build`, `--em.directories` CLI flags removed | — | Use `-c` / `-c.directories` instead |
| `npmSkipBuildFromSource` removed | ✓ | Replaced by `nativeModules.buildDependenciesFromSource` |
| `buildDependenciesFromSource`, `nodeGypRebuild`, `npmRebuild`, `nativeRebuilder` moved | ✓ | Grouped under `nativeModules` sub-key; `nativeRebuilder` renamed to `rebuildMode` |
| `appImage.systemIntegration` removed | ✓ | Removed automatically |
| `asar-unpack`, `asar-unpack-dir`, `asar.unpackDir`, `asar.unpack` removed | ✓ | Replaced by `asarUnpack` |
| `directories` at package.json root removed | — | Specify under `build.directories` |
| `win.azureSignOptions` index-signature keys moved | ✓ | Extra keys moved into `additionalMetadata` object |
| Toolset env-var overrides removed | — | Replace `APPIMAGE_TOOLS_PATH`, `ELECTRON_BUILDER_NSIS_DIR`, etc. with `toolsets.X: { url, checksum }` |
| Toolset default versions bumped | — | Optionally pin to `"0.0.0"` to restore legacy bundle if needed |

## 1. Update Node.js to >=22.12.0

v27 requires **Node.js 22.12.0 or later**. This is the version where Node's [`require(esm)`](https://nodejs.org/en/blog/release/v22.12.0) support was stabilized (no flags needed), which allows both CJS and ESM consumers to use these packages without any code changes.

Update your local environment:
```bash
# nvm
nvm install 22 && nvm use 22

# fnm
fnm install 22 && fnm use 22
```

Update CI (GitHub Actions):
```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '22'
```

Update your `package.json` engines field if you declare one:
```json
{
  "engines": {
    "node": ">=22.12.0"
  }
}
```

---

## 2. ESM output — what changes for your project

electron-builder packages now ship as native ES modules. On **Node >=22.12.0** both styles continue to work:

**CJS `require()` — still works**
```js
const { build, Platform } = require("electron-builder")
```

**ESM `import` — now the preferred style**
```js
import { build, Platform } from "electron-builder"
```

**TypeScript**
```ts
import { build, Configuration, Platform } from "electron-builder"

const config: Configuration = {
  appId: "com.example.myapp",
  mac: { target: "dmg" },
}

await build({ targets: Platform.MAC.createTarget(), config })
```

If your project uses `"type": "module"` or ESM imports, no changes are needed. If it uses CJS on Node >=22.12, `require()` continues to work as before.

---

## 3. Remove `electronCompile` usage

The `electronCompile` configuration option has been **removed**. `electron-compile` is an unmaintained library with no releases since 2019.

**Remove from your configuration:**
```json5
{
  "build": {
    "electronCompile": true  // ← delete this line
  }
}
```

If your project relies on `electron-compile` for source compilation, migrate to a modern build tool before upgrading to v27:

- **[electron-vite](https://electron-vite.org/)** — recommended; fast, first-class ESM and Electron integration
- **[esbuild](https://esbuild.github.io/)** — extremely fast, minimal configuration
- **[webpack](https://webpack.electron.build/)** — mature, widely used, `electron-webpack` for Electron-specific setup

Quick start with electron-vite:
```bash
npm create electron-vite@latest my-app
```

## 4. `electron-forge-maker-*` plugins

The four Forge maker plugins (`electron-forge-maker-appimage`, `electron-forge-maker-nsis`, `electron-forge-maker-nsis-web`, `electron-forge-maker-snap`) are now native ES modules. The public API is identical — Electron Forge loads them via dynamic `import()` internally, so no config changes are required.

## 5. Programmatic API — no changes

The `build()` function signature, all configuration options, and all exported types are unchanged:

```js
// v26 CJS
const { build } = require("electron-builder")

// v27 CJS — still works on Node >=22.12
const { build } = require("electron-builder")

// v27 ESM — preferred
import { build } from "electron-builder"
```

## 6. `snap` config key removed — use `snapcraft`

The top-level `snap` configuration key is removed. Use `snapcraft` with an explicit `base` field instead.

**Before:**
```json5
{
  "build": {
    "snap": {
      "summary": "My app",
      "base": "core22"
    }
  }
}
```

**After:**
```json5
{
  "build": {
    "snapcraft": {
      "base": "core22",
      "core22": {
        "summary": "My app"
      }
    }
  }
}
```

Supported `base` values: `"core18"`, `"core20"`, `"core22"`, `"core24"`. For new projects, `"core24"` is recommended (requires Electron 25+; Electron 28+ highly recommended).

---

## 7. `GithubOptions.vPrefixedTagName` removed — use `tagNamePrefix`

The `vPrefixedTagName` boolean on `GithubOptions` is removed. Use `tagNamePrefix` to control the tag prefix.

**Before:**
```json5
{
  "build": {
    "publish": {
      "provider": "github",
      "vPrefixedTagName": false
    }
  }
}
```

**After:**
```json5
{
  "build": {
    "publish": {
      "provider": "github",
      "tagNamePrefix": ""      // empty string = no prefix
    }
  }
}
```

To keep the default `v` prefix, simply remove `vPrefixedTagName` — `tagNamePrefix` defaults to `"v"`.

### GitLab tag prefix

`vPrefixedTagName` on `GitlabOptions` is also removed. GitLab releases now always use the `v`-prefixed tag (e.g. `v1.2.3`). There is no replacement option; update your release pipeline if your GitLab tags do not follow this convention.

## 8. `framework`, `nodeVersion`, `launchUiVersion` removed

The `framework`, `nodeVersion`, and `launchUiVersion` configuration fields are removed. Only Electron is supported as a target framework — `proton`/`proton-native` and `libui` support has been removed.

**Remove from your configuration:**
```json5
{
  "build": {
    "framework": "electron",       // ← delete; "electron" was and remains the default
    "nodeVersion": "current",      // ← delete; no effect
    "launchUiVersion": "0.1.0"    // ← delete; no effect
  }
}
```

---

## 9. Removed `PackagerOptions` fields: `devMetadata`, `extraMetadata`

The `devMetadata` and `extraMetadata` fields in the programmatic `PackagerOptions` API are removed (they have thrown `InvalidConfigurationError` since v22).

**Before:**
```ts
await build({
  targets: Platform.MAC.createTarget(),
  devMetadata: { ... },    // ← throws in v22–v26, removed in v27
  extraMetadata: { ... },  // ← throws in v22–v26, removed in v27
})
```

**After:**
```ts
await build({
  targets: Platform.MAC.createTarget(),
  config: {
    extraMetadata: { ... },
  },
})
```

- **`appImage.systemIntegration`** — removed. Desktop integration is handled automatically by [AppImageLauncher](https://github.com/TheAssassin/AppImageLauncher).

The following asar-related config keys are removed (they have thrown since v22). Use `asarUnpack` in all cases.

| Removed key | Replacement |
|---|---|
| `asar-unpack` | `asarUnpack` |
| `asar-unpack-dir` | `asarUnpack` |

## 13. `directories` must be under `build`, not at the package.json root

Specifying `directories` at the root of `package.json` is removed. Move it under the `build` key.

**Before:**
```json
{
  "name": "my-app",
  "directories": { "output": "dist" }
}
```

**After:**
```json
{
  "name": "my-app",
  "build": {
    "directories": { "output": "dist" }
  }
}
```

---

## 14. Native module options grouped under `nativeModules`

Four root-level configuration properties are moved into a new `nativeModules` sub-key, and `nativeRebuilder` is renamed to `rebuildMode`.

**Before (v26):**
```json5
{
  "build": {
    "buildDependenciesFromSource": true,
    "nodeGypRebuild": false,
    "npmRebuild": true,
    "nativeRebuilder": "parallel"
  }
}
```

**After (v27):**
```json5
{
  "build": {
    "nativeModules": {
      "buildDependenciesFromSource": true,
      "nodeGypRebuild": false,
      "npmRebuild": true,
      "rebuildMode": "parallel"
    }
  }
}
```

`npmArgs` is not affected — it controls the package-manager install phase and remains at the root level.

Additionally, `npmSkipBuildFromSource` (which was already deprecated in v26) is removed. It is the logical inverse of `buildDependenciesFromSource`:

```json5
// v26
{ "npmSkipBuildFromSource": true }

// v27 equivalent
{ "nativeModules": { "buildDependenciesFromSource": false } }
```

The `migrate-schema` command handles both transformations automatically.

## 15. `win.azureSignOptions` — extra keys moved to `additionalMetadata`

In v26, `WindowsAzureSigningConfiguration` used an index signature (`[k: string]: string`) that allowed arbitrary extra keys alongside the typed fields. These extra keys were passed as additional arguments to `Invoke-TrustedSigning`.

In v27, the PowerShell-based `Invoke-TrustedSigning` integration is replaced by `signtool.exe /dlib Azure.CodeSigning.Dlib.dll /dmdf metadata.json`. Extra fields must now be placed inside an explicit `additionalMetadata` object, which is forwarded verbatim into `metadata.json`.

**Before (v26):**
```json5
{
  "win": {
    "azureSignOptions": {
      "endpoint": "https://weu.codesigning.azure.net/",
      "codeSigningAccountName": "my-account",
      "certificateProfileName": "my-profile",
      "publisherName": "CN=My Company",
      "ExcludeCredentials": "ManagedIdentityCredential"
    }
  }
}
```

**After (v27):**
```json5
{
  "win": {
    "azureSignOptions": {
      "endpoint": "https://weu.codesigning.azure.net/",
      "codeSigningAccountName": "my-account",
      "certificateProfileName": "my-profile",
      "publisherName": "CN=My Company",
      "additionalMetadata": {
        "ExcludeCredentials": "ManagedIdentityCredential"
      }
    }
  }
}
```

The `migrate-schema` command moves any unrecognized keys into `additionalMetadata` automatically.

## 20. Toolset defaults updated

All built-in binary toolsets have been upgraded to modern default versions in v27. The new defaults include:

| Toolset | v26 default | v27 default | Key improvement |
|---------|-------------|-------------|-----------------|
| `winCodeSign` | `0.0.0` (winCodeSign 2.6.0) | `"1.1.0"` | Windows Kits 10.0.26100.0; modern `signtool.exe` and `osslsigncode` |
| `nsis` | `0.0.0` (NSIS 3.0.4.1, split bundle) | `"1.2.1"` | NSIS 3.12; unified single-archive bundle |
| `appimage` | `0.0.0` (FUSE2 runtime) | `"1.0.3"` | FUSE3-compatible static runtime (runs without host FUSE install) |
| `wine` | `0.0.0` (Wine 4.0.1, macOS only) | `"1.0.1"` | Wine 11.0; macOS + Linux; ia32 via WoW64 |

**No action required** for most projects — the new bundles are drop-in replacements and produce identical output.

**If you hit a regression** introduced by a newer bundle, you can pin back to the legacy bundle by setting the toolset version to `"0.0.0"` in your build config:

```json5
{
  "build": {
    "toolsets": {
      "winCodeSign": "0.0.0",   // restore winCodeSign 2.6.0
      "nsis": "0.0.0",          // restore NSIS 3.0.4.1
      "appimage": "0.0.0",      // restore FUSE2 AppImage runtime
      "wine": "0.0.0"           // restore Wine 4.0.1 (macOS only)
    }
  }
}
```

## 21. Toolset env-var overrides replaced by `ToolsetCustom`

**This is a breaking change if you used env-var toolset overrides.**

The following environment variables that previously let you point electron-builder at a local binary bundle have been **removed**:

| Removed env var | Toolset it controlled |
|---|---|
| `APPIMAGE_TOOLS_PATH` | AppImage build tools (`mksquashfs`, runtime) |
| `LINUX_TOOLS_MAC_PATH` | Linux-tools-mac bundle (`ar`, `lzip`, `gtar`) |
| `CUSTOM_FPM_PATH` | FPM executable |
| `ELECTRON_BUILDER_NSIS_DIR` | NSIS compiler bundle directory |
| `ELECTRON_BUILDER_NSIS_RESOURCES_DIR` | NSIS resources/plugins directory |
| `CUSTOM_NSIS_RESOURCES` | Alternate NSIS resources bundle |
| `ELECTRON_BUILDER_WINE_TOOLSET_DIR` | Wine bundle directory |

### Migration: use `ToolsetCustom`

Replace env-var overrides with the new `ToolsetCustom` config object on the relevant `toolsets` key. The `url` accepts an `https://` URL (downloaded and cached automatically) or a `file://` path (used as-is).

**Remote bundle (URL)**
```json5
{
  "build": {
    "toolsets": {
      "nsis": {
        "url": "https://example.com/my-nsis-bundle-1.0.tar.gz",
        "checksum": "sha256:abc123...",
        "version": "my-custom-1.0"
      }
    }
  }
}
```

**Local archive**
```json5
{
  "build": {
    "toolsets": {
      "wine": {
        "url": "file:///path/to/my-wine-bundle.tar.gz",
        "checksum": "sha256:def456..."
      }
    }
  }
}
```

**Local directory (no checksum required)**
```json5
{
  "build": {
    "toolsets": {
      "appimage": {
        "url": "file:///path/to/my-appimage-tools-dir"
      }
    }
  }
}
```

The bundle must mirror the directory layout of the corresponding built-in bundle. See [electron-builder-binaries/packages](https://github.com/electron-userland/electron-builder-binaries/tree/master/packages) for the expected structure.

Supported archive formats: `.zip`, `.7z`, `.tar.gz`, `.tar.xz`. **Exception for `sevenZip`**: because 7-Zip is used to extract `.7z` and `.tar.xz` archives, a custom `sevenZip` bundle can only be supplied as a `.tar.gz`, `.zip`, or bare `file://` directory.

## 22. Plugin and custom-target authors — `PlatformPackager` API changes

> **This section only affects you if you import from `app-builder-lib` and access the `packager` or `platformPackager` objects directly.** Standard project configurations are unaffected.

### `info: Packager` is now `@deprecated`

`PlatformPackager` has always exposed a public `info: Packager` field that gave access to the whole-build orchestrator. In v27 this field is **deprecated** and will become `protected` in a future major release.

TypeScript will emit a `TS6385` hint anywhere your code chains through `.info.`:

```ts
// deprecated — produces a TS6385 hint
const tmpDir = packager.info.tempDirManager
const meta = packager.info.metadata
await packager.info.emitArtifactCreated(event)
```

### New pass-through getters and methods

All commonly-needed properties are now directly accessible on `PlatformPackager` itself. Migrate by dropping the `.info.` chain:

| Before (deprecated) | After |
|---|---|
| `packager.info.tempDirManager` | `packager.tempDirManager` |
| `packager.info.metadata` | `packager.metadata` |
| `packager.info.framework` | `packager.framework` |
| `packager.info.cancellationToken` | `packager.cancellationToken` |
| `packager.info.repositoryInfo` | `packager.repositoryInfo` |
| `packager.info.relativeBuildResourcesDirname` | `packager.relativeBuildResourcesDirname` |
| `packager.info.stageDirPathCustomizer` | `packager.stageDirPathCustomizer` |
| `packager.info.areNodeModulesHandledExternally` | `packager.areNodeModulesHandledExternally` |
| `packager.info.isPrepackedAppAsar` | `packager.isPrepackedAppAsar` |
| `packager.info.appDir` | `packager.appDir` |
| `packager.info.getWorkspaceRoot()` | `packager.getWorkspaceRoot()` |
| `packager.info.emitArtifactBuildStarted(e)` | `packager.emitArtifactBuildStarted(e)` |
| `packager.info.emitArtifactBuildCompleted(e)` | `packager.emitArtifactBuildCompleted(e)` |
| `packager.info.emitArtifactCreated(e)` | `packager.emitArtifactCreated(e)` |
| `packager.info.emitMsiProjectCreated(p)` | `packager.emitMsiProjectCreated(p)` |
| `packager.info.emitAppxManifestCreated(p)` | `packager.emitAppxManifestCreated(p)` |

The getters already on `PlatformPackager` in v26 are unchanged (`config`, `projectDir`, `buildResourcesDir`, `packagerOptions`, `appInfo`, `debugLogger`).

`platformSpecificBuildOptions` is now `protected`. External consumers should use the two new public helpers instead:

| Access pattern | v27 replacement |
|---|---|
| `packager.platformSpecificBuildOptions` (direct read) | `packager.platformOptions` |
| `deepAssign({}, packager.platformSpecificBuildOptions, config.X)` | `packager.getOptionsForTarget<T>("X")` |

`platformOptions` is a typed getter that returns the same platform-level config object. `getOptionsForTarget<T>(key)` performs the standard merge of platform options with the named per-target key (e.g. `"appx"`, `"msi"`) and returns the result as `T`.

## Additional notes

This section documents design decisions and internal changes made during the v27 release for users who want full transparency about what changed under the hood.

### Why native ESM?

`electron-builder` has historically shipped CommonJS. The move to native ESM in v27 was driven by the ecosystem: major dependencies (`chalk`, `figures`, `ora`, etc.) have dropped CJS support, and Node.js 22.12's stabilized `require(esm)` makes it safe to ship ESM without breaking CJS consumers. The minimum version was set to 22.12.0 specifically because earlier Node.js versions require `--experimental-require-module` to `require()` an ESM package.

### Why move native-module options under `nativeModules`?

In v26, options like `buildDependenciesFromSource`, `nodeGypRebuild`, `npmRebuild`, and `nativeRebuilder` were scattered at the root of the `Configuration` object alongside unrelated properties. They were grouped under a `nativeModules` sub-key for the same reason that `directories` and `toolsets` are sub-keys: related options should be co-located. The rename `nativeRebuilder → rebuildMode` makes the field name consistent with other enum-style selectors in the config schema (`"sequential" | "parallel"` is a "mode", not a "rebuilder").

### Why replace `[k: string]` in `WindowsAzureSigningConfiguration`?

In v26, `WindowsAzureSigningConfiguration` used a TypeScript index signature to allow arbitrary extra keys to be forwarded to `Invoke-TrustedSigning`. The v27 Azure integration switches from PowerShell to `signtool.exe /dlib /dmdf`, which reads a `metadata.json` file. An explicit `additionalMetadata: Record<string, string>` field is cleaner than an index signature because it makes the intent visible in IntelliSense and prevents accidental shadowing of the typed fields.

### Why is `snap` a manual migration?

The `snap → snapcraft` migration requires restructuring options under a `base`-named sub-key (e.g., `core22`). Automating this requires inferring the user's `base` value and deciding what to do when it is absent. Because `snap` configurations are highly project-specific and the risk of silently producing a broken config is high, `migrate-schema` issues a warning but does not attempt to rewrite the structure.

### `migrate-schema` serialization caveats

- **JSON5 → JSON**: JSON5 supports comments and trailing commas. When `migrate-schema` rewrites a `.json5` file it produces valid JSON (no comments, standard quoting). The command prints a warning when this happens. If preserving comments is important, apply the changes manually.
- **TOML**: The `toml` npm package is read-only (parse only). `migrate-schema` detects TOML configs, prints the required changes, and exits without writing.
- **YAML**: `js-yaml` preserves key order when round-tripping but does not preserve comments.
- **Programmatic configs** (`.js`, `.ts`, `.cjs`, `.mjs`): Cannot be safely transformed — the command prints manual steps instead.

### `PlatformPackager.info` deprecation strategy

`PlatformPackager` previously exposed a public `info: Packager` field that gave plugin authors access to the full build orchestrator. This created a hard coupling: any internal `Packager` refactor became a breaking change for all plugins. In v27, direct pass-through getters and methods were added to `PlatformPackager` for the properties that plugin authors actually need, and `info` was marked `@deprecated`. This is a soft deprecation — `info` still works in v27 but TypeScript emits a `TS6385` hint. It will become `protected` in a future major release.

### `toolsets` config (new in v27)

v27 ships updated binary toolset bundles: a new Windows Kits bundle (with Azure Trusted Signing DLib support), a new NSIS 3.x bundle, and AppImage FUSE3 support. Users who need to stay on the v26 toolset behavior for compatibility can pin any toolset to `"0.0.0"` in their config:

```json5
{
  "toolsets": {
    "winCodeSign": "0.0.0",   // use v26 Windows Kits
    "nsis": "0.0.0",          // use NSIS 3.0.4.1
    "appimage": "0.0.0",      // use FUSE2-based AppImage
    "wine": "0.0.0"           // use Wine 4.0.1
  }
}
```

This escape hatch is intended as a short-term workaround while you upgrade. The `"0.0.0"` alias may be removed in a future major release.
