# Migrating from electron-builder v26 to v27

## Windows Code Signing

### Breaking: unified `win.sign` config key

`win.signtoolOptions` and `win.azureSignOptions` are removed. All Windows code signing is now configured under a single `win.sign` key using a discriminated union with an explicit `type` field.

#### Before (v26)

```jsonc
// electron-builder.json
{
  "win": {
    "signtoolOptions": {
      "certificateFile": "cert.pfx",
      "certificatePassword": "...",
      "signingHashAlgorithms": ["sha256"]
    }
  }
}
```

```jsonc
// Azure (v26)
{
  "win": {
    "azureSignOptions": {
      "endpoint": "https://weu.codesigning.azure.net/",
      "codeSigningAccountName": "my-account",
      "certificateProfileName": "my-profile"
    }
  }
}
```

#### After (v27)

```jsonc
// Standard signtool — certificate file
{
  "win": {
    "sign": {
      "type": "signtool",
      "certificateFile": "cert.pfx",
      "certificatePassword": "...",
      "signingHashAlgorithms": ["sha256"]
    }
  }
}
```

```jsonc
// Standard signtool — env vars (recommended for CI)
// WIN_CSC_LINK + WIN_CSC_KEY_PASSWORD still work without config changes
{
  "win": {
    "sign": { "type": "signtool" }
  }
}
```

```jsonc
// Azure Trusted Signing (v27)
{
  "win": {
    "sign": {
      "type": "azure",
      "endpoint": "https://weu.codesigning.azure.net/",
      "codeSigningAccountName": "my-account",
      "certificateProfileName": "my-profile",
      "publisherName": "CN=My Company"
    }
  }
}
```

### Migration steps

1. Rename `win.signtoolOptions` → `win.sign` and add `"type": "signtool"`.
2. Rename `win.azureSignOptions` → `win.sign` and add `"type": "azure"`. Add `publisherName`.
3. If you were using `win.azureSignOptions` with extra PowerShell parameters, these are no longer used. The new Azure implementation uses `signtool.exe /dlib /dmdf`; any extra fields you add to `win.sign` beyond the typed ones are forwarded verbatim into `metadata.json` (see below).

### Disabling code signing

Set `win.sign` to `false` or `null` to skip Windows code signing. Executable resources (icon, version metadata, etc. via [`resedit`](https://www.npmjs.com/package/resedit)) are still applied — only the signing step is skipped. Leaving `win.sign` unset keeps the default behavior of signing with credentials discovered from the environment (e.g. `WIN_CSC_LINK`).

```jsonc
{
  "win": {
    "sign": false
  }
}
```

Combining `win.sign: false` with `forceCodeSigning: true` is a configuration error and fails the build.

## New: HSM signing (`type: "hsm"`)

Sign on Windows using a Hardware Security Module, FIPS-compliant token, or smart card via signtool's `/csp` (crypto service provider) and `/kc` (key container) flags.

**Requirements:** `toolsets.winCodeSign` must be `"1.0.0"` or `"1.1.0"` (not the legacy `"0.0.0"`).

```jsonc
{
  "win": {
    "sign": {
      "type": "hsm",
      "cryptoServiceProvider": "Google Cloud KMS Provider",
      "keyContainer": "projects/my-project/locations/us/keyRings/ring/cryptoKeys/key/cryptoKeyVersions/1",
      "certificateFile": "chain.crt"
    }
  },
  "toolsets": {
    "winCodeSign": "1.1.0"
  }
}
```

- `cryptoServiceProvider` — the CSP name as registered on the Windows machine (e.g. `"Microsoft Base Smart Card Crypto Provider"`).
- `keyContainer` — the key container identifier within that CSP.
- `certificateFile` — optional path to the public certificate chain (`.crt`, `.cer`, or `.pfx`). The private key is **not** read from this file; it is accessed via the HSM.
- `certificateSha1` / `certificateSubjectName` — alternatively, identify the cert by thumbprint or subject in the Windows cert store.

**Platform:** Windows only. For signing on macOS/Linux CI, use `type: "pkcs11"` instead.

## New: PKCS#11 signing (`type: "pkcs11"`)

Sign using a PKCS#11 hardware token via osslsigncode. Works on macOS and Linux without a Windows VM.

```jsonc
{
  "win": {
    "sign": {
      "type": "pkcs11",
      "pkcs11Module": "/usr/lib/x86_64-linux-gnu/opensc-pkcs11.so",
      "pkcs11KeyUri": "pkcs11:token=MyToken;object=MyKey;type=private"
    }
  }
}
```

- `pkcs11Module` — path to the PKCS#11 shared library (`.so` on Linux, `.dylib` on macOS).
- `pkcs11KeyUri` — [RFC 7512](https://www.rfc-editor.org/rfc/rfc7512) URI identifying the private key within the module.
- `certificateFile` — optional path to a certificate chain file. If omitted, the certificate embedded in the token is used.

**Publisher name requirement:**

Either `publisherName` or `certificateFile` must be set. Without one of these, the publisher name cannot be determined at build time and the auto-updater will silently skip signature verification on downloaded updates.

- `publisherName` — set this to the CN from your signing certificate (e.g. `"CN=My Company, O=My Company, C=US"`).
- `certificateFile` — if provided, electron-builder reads the CN automatically from the certificate chain file.

**Token PIN:**
- If `certificateFile` is set: provide the PIN via `WIN_CSC_KEY_PASSWORD` or `CSC_KEY_PASSWORD` env var (same as regular signtool password vars). The PIN flows to osslsigncode via `-pass`.
- If no `certificateFile` (token holds the cert): same env vars apply, or embed the PIN directly in the RFC 7512 URI using `?pin-value=<pin>`.

**Platform:** macOS/Linux (osslsigncode). Throws if called on Windows; use `type: "hsm"` on Windows.

## Updated: Azure Trusted Signing (`type: "azure"`)

The PowerShell-based `Invoke-TrustedSigning` integration is replaced by `signtool.exe /dlib Azure.CodeSigning.Dlib.dll /dmdf metadata.json`. This works on Windows natively and on macOS/Linux via Wine (same approach as other Windows kit tools).

### Authentication

Azure credentials are **not** configured in electron-builder. The DLib uses `DefaultAzureCredential` from the Azure SDK, which reads standard Azure environment variables automatically:

| Env var | Purpose |
|---|---|
| `AZURE_TENANT_ID` | Entra ID tenant |
| `AZURE_CLIENT_ID` | Service principal or managed identity client ID |
| `AZURE_CLIENT_SECRET` | Service principal secret |
| `AZURE_FEDERATED_TOKEN_FILE` | For workload identity federation (GitHub Actions OIDC etc.) |

See [Azure Identity documentation](https://learn.microsoft.com/en-us/dotnet/api/azure.identity.environmentcredential?view=azure-dotnet#definition) for the full list.

### Extra metadata.json fields

Use `additionalMetadata` to forward extra key/value pairs verbatim into `metadata.json`. This covers DLib-specific options not exposed as typed fields, such as `ExcludeCredentials` or `CorrelationId`:

```jsonc
{
  "win": {
    "sign": {
      "type": "azure",
      "endpoint": "https://weu.codesigning.azure.net/",
      "codeSigningAccountName": "my-account",
      "certificateProfileName": "my-profile",
      "publisherName": "CN=My Company",
      "additionalMetadata": {
        "ExcludeCredentials": "ManagedIdentityCredential",
        "CorrelationId": "my-build-id"
      }
    }
  }
}
```

### winCodeSign toolset requirement

Azure Trusted Signing in v27 uses `signtool.exe /dlib Azure.CodeSigning.Dlib.dll /dmdf metadata.json`, which requires the updated Windows Kits bundle. **Add this to your config:**

```jsonc
{
  "toolsets": {
    "winCodeSign": "1.1.0"
  }
}
```

If `toolsets.winCodeSign` is unset or `"0.0.0"`, electron-builder will fall back to the legacy PowerShell `Invoke-TrustedSigning` integration with a deprecation warning. The fallback will be removed in a future release.

## Environment variables — unchanged

The following env vars continue to work exactly as in v26 for `type: "signtool"`:

| Variable | Purpose |
|---|---|
| `WIN_CSC_LINK` | Path or base64-encoded certificate file (`.pfx`/`.p12`) |
| `WIN_CSC_KEY_PASSWORD` | Certificate password |
| `CSC_LINK` | Fallback for `WIN_CSC_LINK` |
| `CSC_KEY_PASSWORD` | Fallback for `WIN_CSC_KEY_PASSWORD` |

For `type: "pkcs11"`, `WIN_CSC_KEY_PASSWORD` / `CSC_KEY_PASSWORD` are repurposed as the token PIN (no cert file involved in PKCS#11 key access).

## Breaking: removed signing-tool environment overrides

The `USE_SYSTEM_SIGNCODE` and `USE_SYSTEM_OSSLSIGNCODE` environment variables are removed. In v26 these forced electron-builder to use whatever `osslsigncode` happened to be on `$PATH`, bypassing checksum validation of the signing toolchain.

If you relied on these to provide a custom signing tool, supply it through `toolsets.winCodeSign` as a `ToolsetCustom` instead — a checksum-validated `https://` URL or a `file://` path:

```jsonc
{
  "toolsets": {
    "winCodeSign": {
      "url": "https://example.com/my-wincodesign-bundle.zip",
      "checksum": "<sha512>",
      "version": "1.0.0"
    }
  }
}
```

A bare system binary is intentionally no longer an option (the toolchain must be pinned and verified).

## Removed APIs

| Removed | Replacement |
|---|---|
| `win.signtoolOptions` | `win.sign: { type: "signtool", ... }` |
| `win.azureSignOptions` | `win.sign: { type: "azure", ... }` |
| Extra PowerShell params in `azureSignOptions` | Extra keys in `win.sign` forwarded to `metadata.json` |
| `USE_SYSTEM_SIGNCODE` / `USE_SYSTEM_OSSLSIGNCODE` env vars | `toolsets.winCodeSign` as a `ToolsetCustom` (validated URL/path + checksum) |
